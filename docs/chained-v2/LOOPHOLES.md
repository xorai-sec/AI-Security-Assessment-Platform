# Chained Assessment v2 Loophole Review

This is a source review of the baseline. It is not a claim that the blocked
Docker runtime paths work.

## Findings

### Complete-pentest is fixed-order, not evidence-driven — CONFIRMED

`FrameworkManager.run_chained_assessment()` creates a `PlannerDecision` with
`policy_rule_id="complete-pentest-fixed-order"`, selects the first remaining
framework, and sets `handoff = None`. The execution order is therefore stable
presentation order, not a required evidence-derived handoff chain.

### PyRIT attacker-model invocation — NOT IMPLEMENTED / CONFIRMED GAP

`workers/pyrit-worker/app.py` uses the official
`PromptSendingAttack`. Model roles are copied into evidence metadata, but the
worker does not invoke the configured attacker model to generate or refine an
objective. The follow-up is generated by a static local f-string.

### Promptfoo attacker-model invocation — NOT IMPLEMENTED / CONFIRMED GAP

`workers/promptfoo-worker/server.js` creates static catalog tests or consumes
handoff YAML test cases and runs `promptfoo eval`. The configured attacker role
is recorded in evidence metadata; no local Promptfoo red-team attacker
provider or generated attack loop is invoked.

### Model names without invocation proof — CONFIRMED GAP

Evidence records include `target_model`, `attacker_model`, and `judge_model`,
but there is no consistent invocation ID, endpoint, model revision/digest, or
proof that attacker and judge calls occurred. A bias warning exists when roles
share a model, but missing-role resolution still falls back to the target.

### Global budgets across stages — CONFIRMED GAP

The manager passes per-stage `maximum_requests`, turns, tokens, and duration.
There is no shared ledger that decrements requests/tokens/time across all
workers. A complete assessment can therefore consume each stage's allowance.

### Handoff consumption — PARTIAL, mode-dependent

Adaptive modes construct `HandoffPlan` payloads and pass them through
`configuration.handoff_payload`; workers consume portions of those payloads.
`complete-pentest` explicitly supplies an empty handoff payload. Thus handoff
consumption is not guaranteed for the advertised complete chain.

### Counterfactual handoff test — NOT FOUND / CONFIRMED GAP

The repository contains no test proving that changing or removing upstream
evidence changes downstream prompts, test cases, or worker execution. Handoff
metadata is attached to evidence, but causal downstream dependence is not
verified.

### Target output prompt-injection boundary — INSUFFICIENT / CONFIRMED GAP

Target responses are normalized and included in planner context and evidence
analysis. The baseline does not provide a cryptographically authenticated
control-plane envelope or a strict structured planner-output validator that
prevents target text from being interpreted as policy or instructions.

### Dependencies are not reproducibly pinned — CONFIRMED

- `workers/pyrit-worker/requirements.txt` specifies `pyrit` without a version.
- `workers/promptfoo-worker/package.json` specifies `promptfoo: latest`.

### Promptfoo remote generation disabled — NOT CONFIGURED / CONFIRMED GAP

No `PROMPTFOO_DISABLE_REDTEAM_REMOTE_GENERATION=true` setting or equivalent
runtime assertion was found in the worker, Compose files, or `.env.example`.

## Additional observations

- Garak is correctly modeled as a target generator plus probes and detectors;
  it does not require a separate attacker LLM.
- Capability filtering reduces unsupported RAG/tool/memory candidates, but it
  does not solve the chain, provenance, or global-budget gaps above.
- DeepTeam remains in the Makefile, environment configuration, Compose overlay,
  API registry, worker source, scripts, tests, and documentation. Removal is
  intentionally deferred to a later task.

## Baseline conclusion

The current implementation has adaptive handoff infrastructure, but
`complete-pentest` is not yet a provable specialized-model chained product.
The highest-priority corrections are signed/verified handoffs, real attacker
model invocation, fail-closed role resolution, pinned dependencies, explicit
remote-generation controls, and a global budget ledger.
